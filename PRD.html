2. Цілі і не-цілі

Цілі (що система ПОВИННА робити)

  • Менеджер вводить KPI через веб-форму — дані зберігаються одразу і надійно
  • Кожен індикатор має вагу — система рахує EarnedPoints / PossiblePoints
  • TotalKPI% = TeamKPI%×0.8 + IndividualKPI%×0.2 (Admin = 100% Team)
  • Дашборд: перегляд KPI по датах, кімнатах, стафу
  • Фільтр порушень: які індикатори порушуються найчастіше і ким
  • Дані ніколи не губляться — немає черги, немає синку, немає зовнішніх сервісів
  • Система працює офлайн (локальний PHP+MySQL на хостингу)

Не-цілі (що V3 НЕ робить)

  • ❌ Не зберігає копію в Google Sheets (можна додати пізніше як опцію)
  • ❌ Не має мобільного додатку (responsive веб-інтерфейс достатньо)
  • ❌ Не має системи авторизації з ролями (проста http-auth або нічого — V3.1)
  • ❌ Не має real-time оновлень (достатньо refresh сторінки)

6. PHP REST API — Ендпоінти

Всі ендпоінти повертають JSON. Вхідні дані — JSON у тілі POST-запиту. Файлова структура: api/kpi.php, api/reports.php, api/config.php

api/kpi.php — Введення KPI

POST /api/kpi.php?action=saveTeam
Зберегти командні KPI для одного слоту. Одна транзакція — або всі рядки або нічого.
// Вхідний JSON: { "batch_id": "B-1776634605910", "date": "04-20-2026", "room_code":"Pink", "slot_code":"09AM_11AM", "present_staff_ids": ["st_tamar", "st_alla"], "submitted_by": "manager_1", "items": [ { "indicator": "Cleanliness...", "check": "yes", "comment": "" }, { "indicator": "Temperature...", "check": "no", "comment": "too cold" } ] } // Відповідь: { "ok": true, "saved": 8 } // 8 рядків (4 ind × 2 staff)
POST /api/kpi.php?action=saveIndividual
Зберегти індивідуальні KPI для одного стафу.
GET /api/kpi.php?action=checkDuplicate&batch_id=B-123&staff_id=st_tamar&slot_code=09AM_11AM
Перевірка перед збереженням — чи вже є такий запис.

api/reports.php — Дашборд і аналітика

GET /api/reports.php?action=summary&from=2026-04-01&to=2026-04-30&room=Pink
KPI% по стафу за період. SQL GROUP BY — відповідь за 10–50мс.
// SQL всередині (один запит, без PHP-циклів): SELECT staff_id, ROUND(SUM(CASE WHEN scope='Team' THEN earned_points ELSE 0 END) / SUM(CASE WHEN scope='Team' THEN possible_points ELSE 0 END) * 100, 2) AS team_kpi, ROUND(SUM(CASE WHEN scope='Individual' THEN earned_points ELSE 0 END) / NULLIF(SUM(CASE WHEN scope='Individual' THEN possible_points ELSE 0 END),0) * 100, 2) AS ind_kpi FROM kpi_entries WHERE room_code = 'Pink' AND entry_date BETWEEN '2026-04-01' AND '2026-04-30' GROUP BY staff_id;
GET /api/reports.php?action=violations&from=2026-04-01&to=2026-04-30
Топ порушень: які індикатори найчастіше check=no, по кімнатах і стафу.
// SQL запит для топ-порушень: SELECT room_code, indicator, category, COUNT(*) AS violations, COUNT(DISTINCT entry_date) AS days_violated FROM kpi_entries WHERE check_value = 0 AND entry_date BETWEEN '2026-04-01' AND '2026-04-30' GROUP BY room_code, indicator, category ORDER BY violations DESC LIMIT 20;
GET /api/reports.php?action=daily&date=2026-04-20&room=Green
Детальний денний звіт по кімнаті — всі записи за день.
GET /api/reports.php?action=monthly&year=2026&month=4
Місячний зведений KPI — по всіх стафах і кімнатах.

api/config.php — Конфігурація

GET /api/config.php
Повертає всі конфіги для UI: кімнати, слоти, стаф, індикатори з вагами.
POST /api/config.php?action=updateIndicator
Оновити вагу або статус індикатора.

9. Математика KPI — Специфікація розрахунку

Правила розрахунку (незмінні)

-- Для кожного запису в kpi_entries: earned_points = weight (якщо check_value = 1) earned_points = 0 (якщо check_value = 0) possible_points = weight (завжди) -- Team KPI для одного стафу за період: TeamKPI% = SUM(earned_points WHERE scope='Team') / SUM(possible_points WHERE scope='Team') × 100 -- Individual KPI: IndKPI% = SUM(earned_points WHERE scope='Individual') / SUM(possible_points WHERE scope='Individual') × 100 (якщо SUM(possible) = 0 → IndKPI% = NULL, не 0) -- Total KPI: IF кімната = Admin: TotalKPI% = TeamKPI% ELSE IF SUM(possible WHERE scope='Individual') = 0: TotalKPI% = TeamKPI% ← не ×0.8, повна вага! ELSE: TotalKPI% = TeamKPI%×0.8 + IndKPI%×0.2

Важливо: Team записи зберігаються ПО КОЖНОМУ стафу

Коли менеджер вводить Team-індикатор при присутніх [st_tamar, st_alla, st_malahat], система записує 3 окремих рядки — по одному для кожного стафу. Кожен отримує однаковий earned/possible. НЕ ділити на кількість людей.
-- Team: 4 присутніх, індикатор weight=2, check=yes -- Записуємо 4 рядки: (batch_id=B-123, staff_id='st_tamar', scope='Team', earned=2, possible=2) (batch_id=B-123, staff_id='st_alla', scope='Team', earned=2, possible=2) (batch_id=B-123, staff_id='st_malahat', scope='Team', earned=2, possible=2) (batch_id=B-123, staff_id='st_filiz', scope='Team', earned=2, possible=2) -- Кожен стаф має свій окремий рядок → легко фільтрувати по staff_id
