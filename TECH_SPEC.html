3. Архітектурні рішення (ADR)

Architecture Decision Records — пояснення чому обрано певний підхід. Змінювати тільки з явного рішення.

ADR-01: Денормалізація weight/category/indicator в kpi_entries — ЗАЛИШАЄТЬСЯ

Рішення: у таблиці kpi_entries зберігаємо weight, category, indicator_text як snapshot — копію на момент введення, навіть якщо конфіг пізніше зміниться.

Чому: це аудиторська система. Якщо вага індикатора "Cleanliness" зміниться завтра з 2 на 3 — записи від вчора мають відображати вагу 2, яка діяла тоді. Зберігання тільки indicator_id FK означало б, що перерахунок завтра дасть інший KPI% за вчорашній день — це руйнує аудиторський слід.

Компроміс: додаємо indicator_id INT NULL як soft-посилання для звітів, але розрахунки завжди базуються на snapshot-полях (weight, earned_points, possible_points).

ADR-02: checkDuplicate залишається як UX-ендпоінт, але не як захист

Рішення: checkDuplicate залишається для показу попередження в UI ("цей слот вже збережено!"). Але єдиний захист від дублікатів — UNIQUE KEY + INSERT IGNORE в БД.

Чому: race condition між GET checkDuplicate і POST saveBatch теоретично можливий, але при одному менеджері на кімнату — малоймовірний. UNIQUE KEY гарантує, що навіть при race condition дублікат не пройде в БД. UX-попередження — зручність, не безпека.

ADR-03: present_staff виноситься в окрему таблицю kpi_batch_presence

Рішення: замість CSV-рядка "st_tamar,st_alla" у kpi_entries — окрема таблиця kpi_batch_presence(batch_id, staff_id).

Чому: CSV в колонці не можна форкнути через SQL JOIN, не можна перевірити цілісність FK, LIKE '%staff_id%' — повільно. Це антипаттерн без виправдань.

ADR-04: Один endpoint saveBatch замість трьох послідовних fetch

Рішення: frontend відправляє одним POST весь batch: team items + individual items для всіх стафів. PHP-бекенд обробляє в одній транзакції.

Чому: три послідовних fetch = 3 окремих транзакції. Якщо мережа падає між 2-м і 3-м — частина стафу збережена, частина ні. Одна транзакція = або все або нічого.
