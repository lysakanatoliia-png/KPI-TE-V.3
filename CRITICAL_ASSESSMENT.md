# Критична оцінка проєкту KPI-TE-V.3

**Дата оцінки:** 2026-04-20  
**Обсяг:** аналіз наявної документації (`PRD.html`, `TECH_SPEC.html`, `README.md`).

## Короткий висновок

Проєкт має **сильну бізнес-мотивацію** і добре структуровану специфікацію для швидкого старту розробки. Водночас у поточному вигляді це переважно **документаційний артефакт без реалізованого коду**, а частина технічних рішень створює ризики для безпеки, якості даних і масштабування.

---

## Що добре

1. **Чітко сформульована причина переходу на V3.**
   - Прямо зафіксовані болі V1/V2 (втрата даних, timeout-и, залежність від GAS/Sheets).
   - Є явний фокус на надійності зберігання через MySQL і транзакції.

2. **Зрозумілий функціональний обсяг продукту.**
   - Формула KPI, ролі користувачів, базові звіти й фільтри описані достатньо конкретно.
   - Виділено `must/should` пріоритети для форм і дашборду.

3. **Добра початкова архітектурна декомпозиція.**
   - Розділення на `form`, `dashboard`, `admin` і окремі API-ендпоїнти — правильний baseline.
   - Є орієнтація на простий деплой без важких залежностей.

4. **Продумана схема даних на базовому рівні.**
   - Передбачені індекси для звітності (`entry_date`, `(room_code, entry_date)`, `(staff_id, entry_date)`), що корисно для аналітики.

---

## Що погано / ризики

1. **Немає реалізації — лише документація.**
   - У репозиторії відсутні PHP/JS/SQL файли, які заявлені в техспеці.
   - Це означає, що зараз проєкт не можна протестувати end-to-end і підтвердити працездатність заявленої архітектури.

2. **Суперечність у темі безпеки.**
   - Заявляється можливість "http-auth або нічого" як не-ціль у V3, що критично слабко для системи з персональними KPI.
   - Одночасно в прикладі API виставлено `Access-Control-Allow-Origin: *`, що без суворої auth-моделі підвищує ризики несанкціонованого доступу.

3. **Модель `UNIQUE KEY (batch_id, staff_id, indicator(200))` потенційно крихка.**
   - Префіксний індекс по `indicator(200)` може дати колізії для довгих/схожих текстів індикаторів.
   - Дублікати краще контролювати через стабільний технічний ідентифікатор індикатора (`indicator_id`) замість тексту.

4. **Надмірна залежність від текстових полів там, де потрібні довідники/зв’язки.**
   - `present_staff` як CSV-рядок у головній таблиці ускладнює аналітику, валідацію і цілісність даних.
   - У `kpi_entries` зберігається багато денормалізованих атрибутів (`indicator`, `category`, `weight`) — це може бути свідомо, але збільшує ризик неузгодженості при змінах конфігів.

5. **Операційні аспекти описані поверхнево.**
   - Немає чіткої політики backup/restore (RPO/RTO), моніторингу, аудит-логування, retention, disaster recovery.
   - Без цього твердження "дані ніколи не губляться" не підтверджене процесами експлуатації.

6. **Нечіткість твердження про "офлайн".**
   - Документ одночасно говорить про веб-інтерфейс на хостингу і "офлайн" режим, але без локального кешу/синхронізації це не офлайн у класичному сенсі.

---

## Критичний вердикт

- **Сильна сторона:** дуже непоганий PRD/TechSpec для старту MVP.
- **Слабка сторона:** відсутність реалізованої системи та недозакриті high-risk питання (auth, CORS, модель унікальності, операційна надійність).
- **Поточний стан:** це **план проєкту**, а не готовий продукт.

**Оцінка зрілості (MVP readiness): 4/10.**  
Документація якісна, але до production-готовності бракує реального коду, тестів і безпекової/операційної дисципліни.

---

## Що робити далі (пріоритетно)

1. **Заблокувати security-борг до релізу:**
   - Впровадити обов’язкову авторизацію (мінімум Basic Auth + IP allowlist, краще token/session).
   - Прибрати `CORS: *`, замінити на allowlist доменів.

2. **Виправити модель даних для дедуплікації/цілісності:**
   - Використати `indicator_id` у `kpi_entries`.
   - Винести `present_staff` у join-таблицю (`kpi_batch_staff`), без CSV.

3. **Зробити "верифікований MVP":**
   - Реалізувати мінімальні `api/kpi.php`, `api/reports.php`, `form.html`.
   - Додати smoke/integration тести і тест-кейси з PRD.

4. **Операційна готовність:**
   - Додати backup policy, журнал помилок, алерти, чіткий runbook відновлення.

5. **Уточнити термінологію "offline":**
   - Або прибрати тезу, або реалізувати реальний offline-first (локальний queue + sync + conflict resolution).
